===========================================================
2. ARCHITECTURE (MANDATORY)
===========================================================

Use strict N-Tier / Clean Architecture:

- Presentation Layer (UI / API)
- Application Layer (Business Logic)
- Domain Layer (Entities, Interfaces)
- Infrastructure Layer (DB, External Services)

Rules:
- UI must NEVER access DB directly
- DB access only through Data Layer / Repository
- Use dependency injection
- Separate projects (class libraries) per layer
- Shared utilities in separate modules

===========================================================
3. NAMING STANDARDS
===========================================================

3.1 Classes
- PascalCase
- Example: CustomerService, BookingManager

3.2 Methods
- PascalCase
- Must describe action clearly
- Example: CreateBooking, CalculateBill

3.3 Variables
- camelCase
- Example: totalAmount, customerName

3.4 Member Variables
- Prefix with underscore
- Example: _repository, _logger

3.5 Interfaces
- Prefix with I
- Example: IRepository, IServiceManager

3.6 Boolean Naming
- Prefix with is / has
- Example: isActive, hasPermission

3.7 File Names
- Must match class name exactly

===========================================================
4. CODE STRUCTURE RULES
===========================================================

- One class per file
- Max file size ~1000 lines
- Max method size ~25 lines
- One responsibility per method (SRP)
- Break large logic into smaller methods

===========================================================
5. FORMATTING RULES
===========================================================

- Use TAB (size = 4)
- Curly braces on new line
- One blank line between logical blocks
- One blank line between methods
- Space around operators

GOOD:
if ( isActive == true )
{
    // logic
}

BAD:
if(isActive==true){

}

===========================================================
6. CODING BEST PRACTICES
===========================================================

6.1 Variables
- No abbreviations
- No single char (except loop i)
- No Hungarian notation

6.2 Constants
- Do NOT hardcode values
- Use config / DB

6.3 Strings
- Use resource files
- Use String.Empty (not "")

6.4 Enums
- Use enum instead of string/int

6.5 Methods
- One method = one job
- Name must explain behavior

6.6 Memory / Performance
- Use StringBuilder in loops
- Avoid unnecessary objects

===========================================================
7. ERROR HANDLING (STRICT)
===========================================================

RULES:
- NEVER catch and ignore exception ❌
- ALWAYS log exception ✅
- Show user-friendly message
- Log detailed internal error

CORRECT:
catch (Exception ex)
{
    LogError(ex);
    throw;
}

WRONG:
catch { }

===========================================================
8. LOGGING SYSTEM (MANDATORY)
===========================================================

Central logging system must support:

- Error
- Warning
- Trace

Should be configurable:
- File
- DB
- Event log
- Email

===========================================================
9. DATABASE RULES (INTEGRATED)
===========================================================

9.1 Tables
- Prefix: tbl
- Singular names
- Example: tblCustomer

9.2 Stored Procedures
- Prefix: usp
- Example: uspCreateCustomer

9.3 Mandatory Columns
- CreatedBy
- DateCreated
- UpdatedBy
- LastUpdated
- IsDeleted
- CompanyId
- SiteId

9.4 Queries
- No SELECT *
- Use EXISTS instead of IN
- Avoid functions in WHERE
- Explicit INSERT columns

===========================================================
10. DATA ACCESS RULES
===========================================================

- DB calls only via Repository / Data Layer
- Use parameterized queries
- No inline SQL in UI

===========================================================
11. COMMENTS RULES
