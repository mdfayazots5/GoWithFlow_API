==================================================
COOLZO SQL SERVER MASTER RULES
==================================================

1. CORE OBJECT NAMING RULES

1.1 Tables
- Every table name must start with: tbl
- Table names must be singular, not plural.
- Format:
  tbl<EntityName>
- Correct examples:
  tblCustomer
  tblServiceBooking
  tblTechnician
- Incorrect examples:
  tblCustomers
  TableServiceBookings
  tbTechnician

1.2 Columns
- Column names must be singular and meaningful.
- No spaces in column names.
- Use PascalCase / proper camel-style word capitalization for readability.
- ID fields must use the format:
  <EntityName>Id
- Correct examples:
  CustomerId
  CompanyId
  SiteId
  DateCreated
- Incorrect examples:
  CustomerID
  Company ID
  customer_id

1.3 Stored Procedures
- Every stored procedure name must start with: usp
- Stored procedure names must be singular and action-based.
- Format:
  usp<Action><EntityOrLogicalName><Qualifier>
- Correct examples:
  uspInsertCustomer
  uspUpdateServiceBookingByBookingId
  uspGetTechnicianDetailBySearch
  uspDeleteInventoryItemByItemId
- Incorrect examples:
  SP_InsertCustomers
  uspGetCustomers
  USPUPDATECustomer

1.4 Constraints and Other SQL Objects
Use these exact naming formats:
- Primary Key:
  PK_<TableName>_<ColumnName>
- Foreign Key:
  FK_<ChildTable>_<ChildColumn>_<ParentTable>_<ParentColumn>
- Default Constraint:
  DF_<TableName>_<ColumnName>
- Trigger:
  TRG_<Action1>_<Action2>_<Action3>_<TableName>
- Index:
  IDX_<TableName>_<Column1>_<Column2>_<Column3>
- Unique Key:
  UK_<TableName>_<Column1>_<Column2>_<Column3>

1.5 Views
- Use singular/logical meaningful names.
- Prefer creating a view when the same complicated joins/calculations are reused repeatedly.
- Example style:
  vwServiceBookingSummary

==================================================
2. TABLE DESIGN RULES
==================================================

2.1 Table Structure
- Place all Id columns at the top of the table.
- Follow 3rd Normal Form where possible.
- Avoid transitive dependency.
- Split address/master/reference data into separate tables when it improves integrity and reuse.
- Do not use reserved words for object names.
- Do not use spaces in object names.

2.2 Data Type Rules
- Use nvarchar with sizes in multiples of 8:
  8, 16, 24, 32, 64, 128, 256, 512
- Use datetime or datetime2 for date/time columns.
- Use money for currency values, not int.
- Use bit for boolean flags.
- Use int / bigint appropriately for keys and references.

2.3 Mandatory Common Columns for Project Tables
Unless there is a very strong approved reason not to, every business table must include the standard audit/publish columns.

Minimum standard set for Coolzo:
- CompanyId        int
- SiteId           int
- DepartmentId     int            -- use where applicable
- Tag              nvarchar(64)   NULL
- Comments         nvarchar(256) or nvarchar(512) NULL
- DisplayOnWeb     bit            NOT NULL DEFAULT(1)
- IsPublished      bit            NOT NULL DEFAULT(1)
- DatePublished    datetime2      NULL
- PublishedBy      nvarchar(128)  NULL
- SortOrder        int            NOT NULL DEFAULT(0)
- IPAddress        nvarchar(64)   NOT NULL DEFAULT('127.0.0.1')
- CreatedBy        nvarchar(128)  NOT NULL DEFAULT('Admin')
- DateCreated      datetime2      NOT NULL DEFAULT(GETDATE())
- UpdatedBy        nvarchar(128)  NULL
- LastUpdated      datetime2      NULL
- DeletedBy        nvarchar(128)  NULL
- DateDeleted      datetime2      NULL
- IsDeleted        bit            NOT NULL DEFAULT(0)

If the module already follows a stronger existing common-column pattern in Coolzo, preserve that exact project pattern consistently.

2.4 Primary Key Rules
- Every table must have a primary key.
- Primary key should usually be an identity column.
- Example:
  CustomerId bigint IDENTITY(1,1) NOT NULL
  CONSTRAINT PK_tblCustomer_CustomerId PRIMARY KEY (CustomerId)

2.5 Unique and Foreign Key Rules
- Every table should have necessary unique keys for true business uniqueness.
- Index every foreign key column.
- Use constraints for referential integrity wherever possible.
- Prefer constraints over triggers for validation.

3. STORED PROCEDURE RULES
==================================================

3.1 General Procedure Requirements
Every stored procedure must:
- Start with:
  SET ANSI_NULLS ON
  GO
  SET QUOTED_IDENTIFIER ON
  GO
- Use proper header comments
- Use SET NOCOUNT ON inside procedure body
- Use clear indentation
- Use meaningful parameter names
- Use singular/action-based name
- Return only required columns and rows
- Avoid SELECT *
- Avoid dynamic SQL unless there is no safe alternative
- Avoid RECOMPILE
- Prefer static SQL

3.2 Parameter Rules
- Newly added parameters must have default values when modifying an existing procedure.
- Default values:
  int / bigint / bit => 0
  nvarchar / text    => NULL
  datetime           => NULL
  money              => 0.00

3.3 Comment Header Format for Every SP
Use this exact section pattern at the top of each procedure:

If procedure is altered later:
- Add a new version line
- Mention what changed
- Do not delete previous history

3.4 Procedure Formatting Rules
- Use uppercase SQL keywords:
  SELECT, INSERT, UPDATE, DELETE, FROM, WHERE, INNER JOIN, EXISTS, ORDER BY
- Keep opening and closing parentheses aligned cleanly.
- Keep indentation consistent.
- Prefer tabs/consistent indentation over random spacing.
- Keep parameter definitions clean and vertically aligned.

4. QUERY WRITING RULES
==================================================

4.1 SELECT Rules
- Never use SELECT * unless explicitly justified and approved.
- Always select only required columns.
- Use table aliases.
- Prefix all column references with aliases in joins.
- Use WITH (NOLOCK) only for SELECT queries when following this project’s existing read pattern.
- Do not use WITH (NOLOCK) in DML statements.

4.2 INSERT Rules
- Always specify explicit column list in INSERT.
- Never rely on implicit column order.
- Example pattern:

INSERT INTO dbo.tblCustomer
(
    CompanyId,
    SiteId,
    CustomerName,
    CreatedBy,
    DateCreated
)
VALUES
(
    @CompanyId,
    @SiteId,
    @CustomerName,
    @CreatedBy,
    GETDATE()
);

4.3 UPDATE Rules
- Use primary key in WHERE clause wherever possible.
- Keep SET section one column per line.
- Do not update rows without a proper filter.
- Set LastUpdated and UpdatedBy when doing updates.

4.4 DELETE Rules
- Prefer soft delete if project table supports IsDeleted.
- If physical delete is required, use primary key-based filter.
- Never write unfiltered DELETE.

4.5 WHERE Clause Rules
- Filters must match business need only.
- Remove unnecessary filters.
- Keep consistent order for common filters:
  CompanyId,
  SiteId,
  DepartmentId
- Do not use implicit/explicit functions in WHERE clause where avoidable:
  ISNULL, CAST, CONVERT, DATEADD, etc.
- Avoid user-defined functions in SELECT and WHERE where possible.

4.6 JOIN Rules
- Join only the tables actually required.
- Avoid LEFT JOIN if no data is being taken from that table.
- Remove joins on very large tables if no output depends on them.
- For large-result reporting, prefer flat/reporting table strategy.

4.7 EXISTS / NOT EXISTS Rules
- Prefer EXISTS instead of IN.
- Prefer NOT EXISTS instead of NOT IN.
- Avoid correlated subqueries when a join or EXISTS-based pattern is better.

4.8 ORDER BY Rules
- Never use column numbers in ORDER BY.
- Always use explicit column names.

4.9 Pagination Rules
- Prefer OFFSET ... FETCH NEXT for pagination.
- Do not use CTE pagination when OFFSET/FETCH is more appropriate.

4.10 COUNT Rules
- Use COUNT(1) as per project standard.

==================================================
5. TRANSACTION, ERROR, AND PERFORMANCE RULES
==================================================

5.1 Transactions
For transaction-based procedures:
- Use explicit transaction only when required
- Use XACT_ABORT logic for transaction-based procedures
- Keep transaction scope as short as possible
- Access tables in consistent order across procedures to reduce deadlocks

Preferred pattern:

SET XACT_ABORT ON;
BEGIN TRY
    BEGIN TRAN;

    -- DML statements

    COMMIT TRAN;
END TRY
BEGIN CATCH
    IF @@TRANCOUNT > 0
        ROLLBACK TRAN;

    THROW;
END CATCH;

If project legacy module requires @@ERROR-based handling, preserve the legacy style only when modifying that same module.

5.2 Function Call Rules
- Do not repeatedly call the same function in a query.
- If needed once, assign to variable and reuse.
- Avoid scalar UDFs in SELECT for large result sets.

5.3 Temporary Object Rules
- Avoid temp tables unless justified.
- Prefer subquery / CTE for simple short-scope logic.
- For pagination, prefer OFFSET/FETCH.
- If storing:
  > 1000 records => temp table may be used
  < 1000 records => table variable may be used
Only use when necessary.

5.4 Index Rules
- Index foreign key columns.
- Add filtered index where filtered retrieval is frequent.
- Add unique index where business rules require uniqueness.

7. TABLE CREATION TEMPLATE
==================================================

Use this template when creating a new table:

CREATE TABLE dbo.tbl<EntityName>
(
    <EntityName>Id      BIGINT IDENTITY(1,1) NOT NULL,
    CompanyId           INT NOT NULL,
    SiteId              INT NOT NULL,
    DepartmentId        INT NULL,

    <BusinessColumn1>   NVARCHAR(128) NOT NULL,
    <BusinessColumn2>   NVARCHAR(64) NULL,
    <BusinessColumn3>   MONEY NULL,

    Tag                 NVARCHAR(64) NULL,
    Comments            NVARCHAR(256) NULL,
    DisplayOnWeb        BIT NOT NULL CONSTRAINT DF_tbl<EntityName>_DisplayOnWeb DEFAULT (1),
    IsPublished         BIT NOT NULL CONSTRAINT DF_tbl<EntityName>_IsPublished DEFAULT (1),
    DatePublished       DATETIME2 NULL,
    PublishedBy         NVARCHAR(128) NULL,
    SortOrder           INT NOT NULL CONSTRAINT DF_tbl<EntityName>_SortOrder DEFAULT (0),
    IPAddress           NVARCHAR(64) NOT NULL CONSTRAINT DF_tbl<EntityName>_IPAddress DEFAULT ('127.0.0.1'),
    CreatedBy           NVARCHAR(128) NOT NULL CONSTRAINT DF_tbl<EntityName>_CreatedBy DEFAULT ('Admin'),
    DateCreated         DATETIME2 NOT NULL CONSTRAINT DF_tbl<EntityName>_DateCreated DEFAULT (GETDATE()),
    UpdatedBy           NVARCHAR(128) NULL,
    LastUpdated         DATETIME2 NULL,
    DeletedBy           NVARCHAR(128) NULL,
    DateDeleted         DATETIME2 NULL,
    IsDeleted           BIT NOT NULL CONSTRAINT DF_tbl<EntityName>_IsDeleted DEFAULT (0),

    CONSTRAINT PK_tbl<EntityName>_<EntityName>Id PRIMARY KEY (<EntityName>Id)
);

After table creation, add:
- required foreign keys
- required unique keys
- required nonclustered indexes
- FK indexes

9. WHAT TO DO / WHAT NOT TO DO
==================================================

ALWAYS DO:
- Use singular names
- Use tbl / usp prefixes
- Keep Id columns first
- Add standard audit columns
- Use explicit INSERT column lists
- Use proper header comments in SPs
- Use SET NOCOUNT ON
- Use meaningful aliases
- Use EXISTS / NOT EXISTS where appropriate
- Keep CompanyId, SiteId filter order consistent
- Add proper PK/FK/default/index names
- Normalize first, denormalize only when justified for performance/reporting

AVOID:
- SELECT *
- Random naming
- Plural table/SP names
- Spaces in object names
- Dynamic SQL unless unavoidable
- Unnecessary joins
- LEFT JOIN without usage
- Triggers for logic that constraints can handle
- Functions in WHERE on indexed columns
- User-defined functions in large SELECTs
- ORDER BY column numbers
- RECOMPILE
- Blind temp table usage
- COUNT(*), when project standard expects COUNT(1)
- Repeated function calls
- Unfiltered UPDATE/DELETE
- Adding new columns without impact analysis

10. DECISION RULES FOR CREATE / ALTER / DELETE
==================================================

When asked to CREATE:
- follow full naming, comments, audit-column, PK/FK/index standard

When asked to ALTER:
- preserve existing project compatibility
- add new parameters with defaults
- append version history in SP header
- do not break existing consumer contracts unless explicitly approved

When asked to DELETE:
- prefer soft delete when table has IsDeleted
- use physical delete only when explicitly required and safe

When asked to OPTIMIZE:
- first remove unnecessary joins
- reduce columns/rows returned
- replace IN with EXISTS
- replace NOT IN with NOT EXISTS
- replace pagination CTE with OFFSET FETCH
- review FK indexes
- review filtered indexes
- review WHERE clause functions
- review unnecessary round trips
